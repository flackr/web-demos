What is swiping?

Swiping should allow the user to drag or fling some content with their finger.

How does it interact with scrolling?

Users swipe to scroll content, so we need to ensure they can still scroll before swiping.

Scrolling

What if the swipable area has some scrollable content in it?

The user still needs to be able to scroll that content, so we shouldn't immediately consume those swipes to perform the swipe action.

We should scroll as much as possible, and then if the user tries to swipe again they can engage the swipe.

Does this sound familiar? This is how scroll chaining works.

What about outside scrollers?

You should be able to scroll the outer content after you reach the swipe limit.

Scrolling outside of swipable area

What if a swipable area is inside of some scrolling content.

Consider this swipable toggle.

It should swipe first, but if it is already fully swiped then you should be able to scroll the content.

The natural way to handle this is to chain to the ancestor scroller when you swipe beyond the range of the swipable area.

I hope all of this has convinced you that swipable areas function intuitively as scrollers.

Under development

We've shown what works so far, but there is still a lot to be figured out and developed.

Swipe shouldn't slow to the snap point

When snapping to the last snap area, we shouldn't slow down. It should be treated the same as flinging to the end of a scroller.

Many use cases need a backdrop

The dialog element provides a ::backdrop pseudo-element for this purpose. Perhaps a dialog overcroll-area is the right way to accomplish this?

Automatic area activation?

If the overscroll area is a dialog (e.g. menu with a backdrop), or is a button (e.g. pull-to-refresh), perhaps it should perform its activation behavior on coming into view.

Auto dismissal

When you click outside - it feels as though you should dismiss / close the area.

Inerting / focus management

It is surprising to tab into the area when not open. It may also be surprising to tab out of it when it is open, unless this auto dismisses the area.

What's in a name?

It's currently implemented as overscroll, but is there a better term?

  1. overscrollcontainer

    As discussed, swiping needs to be part of the scroll chain, and having these activated after over-scrolling an area feels natural.

  2. swipecontainer

    Swipable areas has been suggested as a name. This maps to the most common gesture used to activate them.

  3. Something else?

We welcome suggestions.
