What are carousels?

Carousels are a common design pattern used on the web and in mobile applications. OpenUI has catalogued a wide range of designs and styles for carousels.

Carousels typically have one or more of the following:

  • Snapping to a particular page.
  • Each page may contain one or multiple items.
  • Next / previous buttons to switch pages.
  • Page markers / shortcuts.

CSS Carousels

The core features of carousels are broken down into a number of features that can be combined to create different experiences.

  • Scroll buttons
  • Scroll markers
  • Stylable fragmentation

Because it is controllable via CSS, you can even switch between a carousel and non-carousel style presentation depending on device form factor.

Scroll buttons

::scroll-button(<direction>) can be styled to create a button which scrolls one page at a time:

Scroll buttons

The ::scroll-button( <scroll-button-direction> ) pseudo-elements are generated on scroll containers when their computed content value is not none. They generate boxes as if they were immediately preceding siblings of their originating element, with content as specified by content.

Directions

Scroll buttons take a direction as an argument.

Physical directions

  • up
  • right
  • down
  • left

Logical directions

  • block-start
  • inline-end
  • block-end
  • inline-start

Smart directions

  • prev
  • next

Selects either the block-start or inline-start ::scroll-button(), whichever's axis has more "scrollable pages" in the originating element

Common styling

Common styles can be set using the * direction. E.g.

::scroll-button(*) { border-radius: 8px; }

What if you can't scroll anymore?

When you reach the maximum possible scroll position, accounting for mandatory snapping, the scroll button becomes disabled.

The style of the disabled button can be customized using the :disabled pseudo-class. E.g.

::scroll-button(*):disabled { opacity: 0.5; }

How far does it scroll

One "page" of content.

We use the same distance as pressing page up / down or the spacebar, which is generally 87.5% of the optimal viewing region, but also applies a maximum overlap between pages of 40px.

Optimal viewing region? What's that?

The optimal viewing region is the region used as the target region for placing things in view of the user, This can be adjusted with the scroll-padding property. E.g.

.scroller { scroll-padding-top: 40px; /* Top 40px is not viewable due to sticky header. */ }

Scroll markers

::scroll-marker creates a navigational shortcut to the establishing element in the ::scroll-marker-group of its ancestor scroll container.

Scroll markers

A scroll marker is any element or pseudo-element with a scroll target.

Creating scroll markers

A ::scroll-marker pseudo-element with a non-none content style, creates a scroll marker pseudo-element, whose generated box is contained by the nearest ancestor scroller's ::scroll-marker-group pseudo-element.

Containing the scroll markers

The scroll-marker-group property on a scrolling container determines whether the ::scroll-marker-group pseudo-element creates a box and whether it is before or after its own box.

I.e. the following scroller creates a ::scroll-marker-group pseudo-element whose box immediately follows the scroller.

.scroller { scroll-marker-group: after; }

Scroll marker group types

In addition to specifying the box tree location, the scroll-marker-group property also specifies the behavior of the scroll marker group. E.g.

.scroller { scroll-marker-group: [ before | after ] [links | tabs ] ;}

Behaviors

  • links

    Links are treated as a list of links. Each link is a tab stop, and all content is accessible regardless of which link is current.

  • tabs

    Tabs are treated as independently activatable sections. All content outside of the current tab is considered inert until current. Tabs can be switched between using the arrow keys once one is selected.

Marking progress

Within a ::scroll-marker-group, exactly one ::scroll-marker matches :target-current. This is the current scroll marker.

The particular position at which a scroll marker is considered to current, is when the target of that scroll marker is more than half-way to its aligned position.

This results in the active scroll marker switching halfway between sections. See the example algorithm in the spec for how this is defined.

Columns

The columns property will split the content in the block direction when it would overflow the container.

The value specifies the number of columns or width of a column, however if there is only 1 column than that column is the full size.

When there are more columns than fit in the current scrollport, they continue to lay out in the inline direction, creating scrollable overflow for a scrolling container.

Styling columns

Now that we have a mechanism for creating scrollable paginated content, we just need to create scroll markers for it. The ::column pseudo-element allows applying limited styling supporting the creation of scroll markers and setting snap alignment. E.g.

::column { scroll-snap-align: center; }

Creating scroll markers

We can also create scroll markers for columns using much the same mechanism:

::column::scroll-marker { content: "Page"; }

Counting

Using CSS counters, we can even number these columns:

::column::scroll-marker { counter-increment: page; content: counter(--page); }

Making a tabs UI

Introduction

Slides should follow the tabs pattern.

Focus

There are a number of implicit focus behaviors:

  • Only the active slide marker is a tab focus stop.
  • Pressing up/down/left/right will move focus between markers and scroll between slides.
  • The marker is a focus navigation scope owner for its contents. This means focus will move from the tab to its contents and vice versa.

Accessibility roles and behaviors

Tabs implicitly implement the expected tab semantics. In particular,

  • The scroll marker group is a role=tablist containing role=tab elements
  • The minimum role of the contents is role=tabpanel
  • Only the active tabpanel is in the accessibility tree.

Conclusion

In summary, these implicit behaviors aim to align with the expectations of a carousel with markers.

Not just CSS, HTML too.

The capabilities of scroll markers and scroll buttons are available for HTML elements as well.

  • page-* command
  • scroll-target-group containing <a> links

These capabilities give access to a similar set of features.

page-* invoker actions

Page commands

The page-<direction> command scrolls the targeted element by one page in the given direction.

Directions

Scroll commands are suffixed with a direction.

Physical directions

  • page-up
  • page-right
  • page-down
  • page-left

Logical directions

  • page-block-start
  • page-inline-end
  • page-block-end
  • page-inline-start

Result

This allows the creation of pure HTML buttons for activating scrolls.

scroll-target-group property

The scroll-target-group property can be set on an element that contains a collection of anchor link references. E.g.

<style> nav { scroll-target-group: auto; } <style> <nav> <a href="#intro">Introduction</a> </nav>

Styling the current target

Once you have established a scroll-target-group, the same algorithm that determines the current scroll marker also determines which link is current.

This link can then be styled with the :target-current pseudo-class:

<style> a:target-current { color: ActiveText; } <style>

Conclusion

In conclusion, the scroll-target-group property is a simple progressive enhancement that can be dropped into existing table of contents, and indicate progress to the user.
